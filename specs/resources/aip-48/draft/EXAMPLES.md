# AIP-48 examples — agentruntimes/v1 worked manifests

Five compositions over the same seven ports. Each one is a complete manifest;
the kernel sees nothing except the port adapters it resolves from `kind`.

## 1. Local swarm — file substrate, two CLI participants

The cut-1 / cut-2 reference. No network. A markdown journal at
`.runtime/conversation.md` is the source of truth for turns. Two example
sub-agents act as participants.

```yaml
schema: agentruntimes/v1
kind: MultiAgentRuntime
id: local-dev-swarm
participants:
  - id: reviewer
    executor: agent-cli
    displayName: Reviewer
    role: ../../.claude/agents/reviewer.md
  - id: migration-writer
    executor: agent-cli
    displayName: MigrationWriter
    role: ../../.claude/agents/migration-writer.md
substrate:
  kind: file
  path: ./conversation.md
dispatcher:
  kind: mention
state:
  kind: fs
  dir: ./state
```

Run with `agentproto run-swarm --manifest .runtime/multi-agent.yaml`.

## 2. Guilde-bridged — substrate is a real conversation

The cut-1 conversation-first reference. The CLI session participates in a
live Guilde conversation as a first-class operator. Humans + Mastra operators
+ this CLI session share one thread.

```yaml
schema: agentruntimes/v1
kind: MultiAgentRuntime
id: guilde-bridge
participants:
  - id: claude-code-operator
    executor: agent-cli
    displayName: Claude Code
    role: ../../.claude/agents/reviewer.md
    meta:
      operatorId: 00000000-0000-0000-0000-000000000001
      guildId: 00000000-0000-0000-0000-000000000002
substrate:
  kind: guilde-mcp
  conversationId: 11111111-1111-1111-1111-111111111111
  asOperatorId: 00000000-0000-0000-0000-000000000001
  fetchLimit: 50
dispatcher:
  kind: mention
state:
  kind: fs
  dir: ./state
```

The substrate adapter wraps Guilde MCP `post_message` and `get_messages`;
`asOperatorId` is the Guilde operator the CLI posts as. The kernel doesn't
know it's Guilde — only the substrate adapter does.

## 3. Mixed-runtime swarm (forward-looking) — one CLI, one Mastra operator, one human

Two participants on the AI side (one Claude Code CLI, one Guilde-hosted
Mastra operator) plus the conversation's human members, all coordinated via
the same dispatcher.

```yaml
schema: agentruntimes/v1
kind: MultiAgentRuntime
id: guilde-mixed
participants:
  - id: claude-reviewer
    executor: agent-cli
    displayName: Claude Reviewer
    meta:
      operatorId: 00000000-0000-0000-0000-000000000010
  - id: stan-publisher
    executor: db-operator
    displayName: Stan Publisher
    meta:
      operatorId: 00000000-0000-0000-0000-000000000020
substrate:
  kind: guilde-mcp
  conversationId: 22222222-2222-2222-2222-222222222222
dispatcher:
  kind: mention
state:
  kind: fs
```

`db-operator` is the executor kind that resolves a Mastra-hosted operator's
turn through Guilde MCP (`runtime.kind === "in-process"` operators). Both
participants share the same dispatcher.

## 4. Capability-routed dispatcher (forward-looking)

When mentions aren't enough — e.g. routing by skill tags rather than display
name.

```yaml
schema: agentruntimes/v1
kind: MultiAgentRuntime
id: capability-routed
participants:
  - id: reviewer
    executor: agent-cli
    displayName: Reviewer
    meta:
      capabilities: ["code-review", "security"]
  - id: migration-writer
    executor: agent-cli
    displayName: MigrationWriter
    meta:
      capabilities: ["database", "schema"]
substrate:
  kind: file
  path: ./conversation.md
dispatcher:
  kind: capability-match
  rubric: skim-and-tag        # adapter-specific config
state:
  kind: fs
```

The `capability-match` dispatcher reads `participant.meta.capabilities` and
picks the best-matching participant for the latest turn. Falls back to no
selection if no participant matches.

## 5. Server-side autonomous loop (forward-looking)

A runtime hosted inside a Guilde server process — no CLI involvement. Same
manifest shape, different adapter wiring.

```yaml
schema: agentruntimes/v1
kind: MultiAgentRuntime
id: guilde-autonomous
participants:
  - id: triager
    executor: in-process-mastra
    meta:
      operatorId: 00000000-0000-0000-0000-000000000030
substrate:
  kind: guilde-mcp
  conversationId: 33333333-3333-3333-3333-333333333333
dispatcher:
  kind: llm-router
  model: claude-sonnet-4-6
state:
  kind: mcp-kv                  # state lives in Guilde, not a local fs
```

`in-process-mastra` is the executor kind for participants whose turn is
generated by a Mastra agent running inside the same server process. State is
also Guilde-hosted, removing the local filesystem dependency entirely. The
manifest looks like the others — the kernel doesn't know.
